01 / 12 기초 문법

RDF — 세 단어 문장

RDF는 진짜 단순합니다. 세 단어 문장 모음이에요. 끝.

민수 ─좋아한다→ 사과
(누가)    (어떻다)    (무엇을)

여러 개를 모으면:

민수 ─좋아한다→ 사과
민수 ─다닌다→ 행복초등학교
사과 ─색깔이→ 빨강
✨ 마법
문장 ①에 "사과", 문장 ③에도 "사과" — 컴퓨터가 *같은 단어*를 보고 이어붙입니다:
"민수가 좋아하는 것의 색깔은 빨강"

왜 #B-3021 같은 이상한 이름? "민수"는 학교에 5명 있을 수 있어요. 컴퓨터가 헷갈리니까 *학번*을 줍니다. 본인 demo의 #B-3021은 무안 한옥마을 건물의 학번이에요.

RDF는 결국 — 세 단어 문장을 잔뜩 적고, 같은 단어 만나면 잇기. 그게 다.

02 / 12 한 층 위로

OWL — 규칙으로 자동 분류

🦉 부엉이 = 현명함의 상징. W3C가 일부러 OOL 대신 OWL로 만든 이름.

OWL은 RDF에 *규칙*을 더하는 언어입니다. RDF가 *사실*이라면, OWL은 *카테고리 규칙*이에요.

RDF (사실)
민수 ─다닌다→ 5학년
콩이 ─종류→ 진돗개
OWL (규칙)
5학년에 다니는 사람은 학생
학생은 모두 사람
진돗개는 개
개는 포유류
포유류는 동물
컴퓨터가 자동으로 알아냄:
민수 → 학생 → 사람
콩이 → 개 → 포유류 → 동물
규칙 한 번 적으면 *모든 인스턴스*에 자동 적용됨.
RDF는 사실, OWL은 규칙. 사실은 사람이 일일이, 규칙은 한 번만. 나머지는 컴퓨터.

03 / 12 가치 명료화

왜 OWL을 쓰는가 — 4가지 이득

한 줄 답: 같은 말 만 번 안 적어도 되고, 컴퓨터가 대신 이해해주니까. 게으름의 정당화.

①

한 번 적으면 *전체*에 적용

한옥 1만 채에 일일이 "전통건축물이야" 안 적어도 됨. 한옥 ⊑ 전통건축물 한 줄이면 끝.

②

컴퓨터가 *틀린 데이터*를 잡아줌

"이 한옥은 *철근콘크리트*다" 잘못 입력 → OWL이 "한옥은 목구조인데 모순!" 자동 감지. 사람이 일일이 검토 안 해도 됨.

③

검색이 *똑똑*해짐

"전통건축물 화재 다 보여줘" → 한옥·종가·사찰 자동 매칭. 매번 길게 적을 필요 없음.

④

시스템 간 자동 매핑

:한옥 owl:sameAs :Hanok 한 줄로, 무안소방서·국가유산청·자체 시스템 모두 자동 호환.

OWL은 "쓰는 사람의 게으름을 정당화하는 언어"입니다. 좋은 게으름 — 큰 데이터에서 사람이 신경 못 쓰는 일관성을 컴퓨터가 알아서 유지.
📚 핵심 질문 "합의"는 어디서 보고 따라하나?

OWL이 합의 명시화라면 — 그 합의는 만드는 게 아니라 발견하는 것입니다. 한국 화재 대응 도메인은 이미 50년치 합의가 쌓여 있어요. 4갈래에서 발견:

  • · 국가유산기본법 — 보존 등급
  • · 재난안전법 — 통보 의무
  • · 소방력 기준 규칙 — 자원 매칭

  • · 건축물 통합정보 (cloud.eais.go.kr)
  • · 국가공간정보 (nsdi.go.kr)
  • · 소방청 통계 (nfa.go.kr)

README가 솔직히 적은 대로, 컬럼 명세만 *참고*했지 학습 단순화로 변형됨. 한옥카페 같은 변종은 임의, EDXL·Brick 같은 표준 import는 없음. 운영 시스템으론 위 표준들을 *진짜 import*해서 그 위에 ontology를 얹어야 합니다. 그게 "합의 명시화"의 진짜 의미.

04 / 12 가장 깊은 본질

조건식과 무엇이 다른가

여기까지 OWL의 능력을 하나씩 부정해 봤습니다. 시나리오 생성도 안 하고, 최적 선택도 안 하고, 확률도 없고, 액션도 룰엔진이 함. 그럼 OWL이 그냥 if/else와 뭐가 다른가? 이게 본 학습 여정에서 가장 깊은 질문이에요.

⭐ 핵심 정의 한 줄 요약
조건문은 단순 규칙,
온톨로지는 도메인에 대한 명제.
학계 정의와 일치 — Tom Gruber (1993):
"An ontology is a specification of a shared conceptualization."
→ specification (= 명제) of a shared conceptualization (= 도메인에 대한)
본 정의는 30년 학계 합의의 한국어 압축판입니다.
같은 일을 두 방식으로
조건식
def should_dispatch_foam(b):
    if b.sub_type in ["한옥", "한옥카페", "종가", "사찰"] \
       and b.fire_risk > 0.8:
        return True

def should_notify_external(b):
    if b.sub_type in ["한옥", "한옥카페", "종가", "사찰"]:
        return True
# 매 함수마다 4개 타입 나열
OWL
# ontology.ttl — 한 번만 선언
:한옥카페 rdfs:subClassOf :Hanok .
:Hanok    rdfs:subClassOf :TraditionalClusterMember .

# rules.yaml — 구체 타입 안 나옴
RULE-002: if a :TraditionalClusterMember and risk > 0.8 then 폼차
RULE-004: if a :TraditionalClusterMember then 통보
새 타입 (전통차마을한옥) 추가 시:
조건식 → 모든 if문 grep + 수정. 누락 시 silent bug.
OWL → ontology.ttl 한 줄. 룰 수정 0줄.
⚖️ 명제 *종류*가 다름
조건식
if x.type == "한옥"
절차적 값 비교.
"x.type 필드 값이 문자열 '한옥'과 같다."
→ 컴퓨터에 던지는 명령
OWL subClassOf
:한옥카페 ⊑ :Hanok
집합론적 포함 명제.
"집합 (한옥카페) ⊆ 집합 (한옥)."
→ 세계의 구조를 서술

전자는 동작 명령, 후자는 존재론적 명제. 카테고리 자체가 달라요.

그 차이가 가져오는 5가지 함의
  1. ①
    Transitive closure가 공짜. 집합론 정리 A⊆B ∧ B⊆C → A⊆C 자동 적용. 한옥카페 → 한옥 → TraditionalClusterMember → Building → Asset, 5단계 자동. 조건식은 매 if마다 풀어 적어야.
  2. ②
    다중 멤버십 자동. x ∈ Hanok이면 자동으로 ∈ TraditionalClusterMember, Building, Asset. 조건식의 x.type은 단 하나의 값.
  3. ③
    명제가 진리값을 가짐. OWL: 한옥카페 ⊑ 한옥 → 참 또는 거짓 (검증 가능). 조건식 if x.type=="한옥" → 진리값 없음, 그냥 실행. 명제니까 논쟁·합의·검증 가능.
  4. ④
    Open World 가능. 안 적힌 것 → "거짓"이 아니라 "모름". 조건식은 매칭 안 되면 자동 거짓 (Closed World). OWL은 모름을 모름으로 표현.
  5. ⑤
    단조성 (Monotonicity). 새 사실 추가해도 기존 결론 안 바뀜. 조건식은 if 순서·else if 추가로 동작이 변할 수 있음. OWL은 증명 가능·재현 가능.
두 가지가 적는 *대상*이 다름
축 조건식 OWL
적는 대상시스템 동작세계의 구조
명제 형태명령 (do this)선언 (X is-a Y)
진리값없음 (실행됨)있음 (참/거짓)
자동 결론적은 만큼만적은 것 + 자동 유도
합의 가능어려움 (코드 리뷰)쉬움 (도메인 전문가가 ttl 직독)
🏛 인류학적 비유 규칙 vs 리추얼

같은 차이를 인류학 어휘로 옮기면 더 선명해집니다:

규칙 (Rule)
  • · "위반하면 처벌" — 외부 강제
  • · 입법자가 작성·고침
  • · 의미는 형식 *밖*에서 부여
  • · = 조건문
리추얼 (Ritual)
  • · 위반 개념 약함 — 형식 자체가 의미
  • · 공동체가 함께 살아냄·변형
  • · 의미가 형식 *안*에 녹아 있음
  • · = 온톨로지의 형식적 측면
⚠️ 단, 완전 일치는 아님

리추얼은 온톨로지보다 풍부해요. 리추얼 = 형식 + 신체적 수행 + 시간성 + 정서·의미. 그중 "형식·합의·분류" 부분만 OWL이 캡처합니다. 신체·시간·정서는 OWL 밖 — ch7 "현존재 차원의 정직"에서 일부러 포기한 부분이고, 그 자리가 LLM의 자리예요.

그래서 코드 리뷰는 기술적 검토지만, 온톨로지 리뷰는 공동체 합의입니다. 도메인 전문가가 ttl을 읽고 "한옥카페는 한옥이 아니라 별도여야 한다"고 말할 수 있는 게 그 차이. OWL은 도메인을 디지털 형식의 리추얼로 보존하는 시도예요 — 정확한 분류보다 합의 가능한 분류가 우선인 이유.

조건식은 "이 시스템이 어떻게 동작하는가"를 적고,
OWL은 "이 도메인이 어떻게 생겼는가"를 적습니다.
규칙은 입법되고, 리추얼은 살아내짐 — 그래서 합의 가능한 명제라는 OWL의 자리가 생겨요.
정직하게 — 작을 땐 조건식이 더 단순

본 데모 스케일(타입 ~10개, 룰 4개)에선 솔직히 조건식이 더 짧고 직관적이에요. OWL의 가치가 진짜로 발현되려면: 분류 차원 수백 개 (의료 SNOMED CT = 35만), 새 타입 추가가 잦음, 다른 시스템과 어휘 교환, 도메인 전문가 검증, 감사·재현성 필요. 본 데모는 이 가치를 학습용으로 미리 체험하는 vehicle이에요.

05 / 12 본질의 차이

시나리오 방식 vs 자동 추론

*결과만 보면 같아 보이지만*, 변화가 닥칠 때 차이가 폭발적으로 벌어집니다.

옷장 비유
📋 시나리오
검은 운동화 → 아빠
빨간 운동화 → 아빠
흰 운동화 → 아빠
갈색 구두 → 아빠
... (30켤레 다)
🦉 추론
"남자 신발 = 아빠 것"
+ 각 신발에 [남자/여자] 태그
새 신발? 태그만 붙이면 자동.
화재 시나리오
건물 5종 × 지역 9종 × 시간 2종 = 90가지:
한옥 + 무안 + 야간 → 폼차2 + 전통팀 + 발주
한옥 + 무안 + 주간 → 폼차2 + 전통팀
한옥 + 일로 + 야간 → 폼차2 + 전통팀 + 발주
... (90가지 다 적기)
vs OWL 룰 4~6개:
한옥 ⊑ 전통건축물
전통건축물 화재 → 폼차2 + 전통팀
야간 화재 → 발주
→ 컴퓨터가 *조합해서* 90가지 자동 생성. 새 건물 추가? 한옥카페 ⊑ 한옥 한 줄. 룰 영원히 안 건드림.
새 케이스
시나리오: O(N)
추론: O(1)
명칭 변경
시나리오: 일괄 수정
추론: URI 한 곳
조합 폭발
시나리오: N×M×K
추론: N+M+K
시나리오는 계산 결과를 미리 저장한 테이블. 추론은 그때그때 계산하는 함수. 결과가 같아 보여도, 변화 비용이 다르다.

묶기 3종 — OWL이 못 하는 것

현존재 차원 — ontology의 정직한 한계

가장 중요한 깨달음입니다. ontology 마케팅의 가장 큰 거짓말이기도 하고요.

Ontology는 세상의 관계의미망을 드러내지 않습니다. 이미 누군가가 가진 해석을 부호화할 뿐이에요.

실제로 일어나는 일:

  1. 1. 누군가가 이미 해석을 가짐 — 소방관은 "한옥엔 화학소방차 쓰면 안 된다"는 *세계 이해*를 ontology 만들기 *전에* 가지고 있어요
  2. 2. ontology는 그 해석을 *기록* — 클래스·관계·룰이 모두 *인간의 결정*
  3. 3. 그래서 ontology는 *드러내는 게 아니라 투사하는* 것 — 발견(theoria)이 아니라 commitment(thesis)
Heidegger 식으로
ontology가 잡는 건 vorhanden (present-at-hand) — 사물을 *속성 가진 객체*로 굳혀서 보는 차원. 의미는 zuhanden (ready-to-hand) — *engagement* 차원에서 일어나요. 소방관이 한옥에 다가갈 때 그건 "builtYear 1923인 객체"가 아니라 *지켜야 할 것·불타고 있는 것·안에 사람 있는 것*으로 engaged됩니다.
:Building/B-3021 a :Hanok 트리플은 이 engagement를 *옮기지 못합니다*. 박제할 뿐.

옮기는 과정에서 *반드시 손실*:

  • · "한옥"의 문화적 두께 — `rdfs:subClassOf`로 못 들어감
  • · 소방관의 직관적 위험감지 — `fireRisk = 0.85`로 환원 안 됨
  • · 보존회장의 애착 — `:contactRole`로 표현 안 됨

이 손실분이 ontology의 비용이에요. 그리고 이 비용은 *피할 수 없습니다*. ontology의 가치는 *발견*이 아니라 명시화에 있어요. 숨겨진 합의를 표면 위로 끌어올려, 토론·감사·수정 가능하게 만드는 것 — 그게 전부입니다. 그게 적은 일은 아니지만, *의미의 발견*은 아닙니다.

JSON + LLM이 OWL을 대체할까?

"Claude에게 JSON으로 룰 적시하면 OWL이랑 똑같지 않아?" 본인 직감이 정확해요. 학습 demo 규모에선 거의 동일. 차이는 production scale에서만 본격 드러납니다.

✅ 같은 효과 (작은 규모)
  • · 클래스 분류 자동 도출
  • · 룰 기반 트리거
  • · 모순 검출
  • · LLM이 *더 나은 점*: 자연어 룰 해석, 컨텍스트 이해, OWL DL 못 하는 추론
❌ OWL이 여전히 이기는 영역
항목 OWL/룰엔진 LLM
속도<1ms1~10초
비용0호출당 $$
결정성보장확률적
감사가능블랙박스
스케일수백만수백만 비쌈
환각없음가능
결정적 사례 — ruledata #how 200ms
15:14:23.001  RULE-001 fires
15:14:23.078  RULE-002 match
15:14:23.112  RULE-003 match
15:14:23.156  RULE-004 match
15:14:23.201  dispatch
200ms 안에 5룰 발동·연쇄. LLM은 1~5초 → 화재 대응엔 부적합.
⚖️ 정직하게 — 본 데모 스케일에선 LLM이 더 빠릅니다

본 데모는 사실 7개 → 14개 자동 도출입니다. 이 정도면 Claude한테 CSV 통째로 넘기고 "추론해" 한 번에 더 짧게 끝나요. 위 표의 OWL 우위(속도·결정성·감사)가 진짜로 필요해지는 건 건물 수만 개·룰 수백 개·SLA 200ms 같은 production scale부터.

그래서 본 데모는 "OWL이 LLM보다 좋다"가 아니라 "OWL이 풀 수 있는 모양은 이런 모양"을 보여주는 게 목적이에요. 도구 선택은 스케일·검증요구·도메인이 결정합니다.

📡 2026-04 참고: 카파시 "LLM Wiki"

Karpathy가 2026년 4월 공개한 "LLM Wiki" 패턴은 본 데모와 정반대 좌표예요. 원본 raw/ 폴더를 LLM이 읽어 상호링크된 마크다운으로 점진 컴파일하는 방식 — RDF·OWL·SPARQL 같은 형식 의미론 없이 LLM에 의미 이해를 맡깁니다.

🦉 본 데모 (OWL 쪽 끝)
사람이 클래스·룰을 명시 → reasoner가 결정적으로 추론. 검증·감사 가능, 변경 비싸다.
📝 LLM Wiki (반대 끝)
자연어 마크다운에 LLM이 의미를 녹여 축적. 변경 싸다, 검증 어렵다.

같은 "LLM × 구조화" 스펙트럼의 양 끝. typed relation·reasoner가 필요한가?가 갈림길입니다. 화재 대응처럼 감사·결정성이 핵심이면 OWL 쪽, 개인 지식관리처럼 유연성이 핵심이면 LLM Wiki 쪽. 커뮤니티는 두 끝을 잇는 GraphRAG·OmegaWiki·Nodus Labs 같은 중간지대를 빠르게 채우는 중이에요.

OWL과 LLM은 경쟁이 아니라 각자 자리예요. 본인의 첫 PNG가 정답이었어요 — 두 엔진이 따로 그려져 있었죠.

4가지 설계 원칙

현존재 차원의 한계를 *부정하지 말고 설계 원칙으로 흡수*하는 게 정공법입니다.

1

"의미를 드러낸다" 카피 금지

데모·문서에서 이 표현 쓰지 마세요. 거짓말입니다. 대신: "공동체가 합의한 해석을 명시화·공유 가능하게 만든다"

2

못 잡는 것 명시

rdfs:comment로 "이 클래스로는 못 잡는 것" 적기. 예: :Hanok엔 *문화적 두께*가 빠졌다고 명시.

3

Provenance 의무화

"누가 이 분류를 결정했나"를 매번 추적. 의미는 권위의 결정이지 데이터의 자명한 모양이 아니라는 걸 항상 가시화.

4

주기적 재해석 절차

1~2년마다 도메인 전문가 다시 모여 "여전히 이 분류가 맞나" 재검토. 정적인 ontology는 죽은 의미.

이 네 원칙이 들어간 ontology 데모는 드물어요. 대부분 1번을 어기고 시작합니다 — "ontology로 의미를 드러내자!"는 슬로건. 본인 데모가 그걸 안 한다면, 그 정직함 자체가 차별점입니다.

본 데모의 한계 — 정직한 진단

"OWL이 준비된 상태이지 살려쓰는 상태는 아닙니다." 학습자에게 이 정직함이 가르침의 일부예요.

현 데모 진단

OWL 어휘 사용 ✅ 사용 중
OWL reasoner 적극 활용 ❌ 시연만
rules.yaml 조건 문자열 매치 (OWL 우회)

왜 이 정도만? 학습 목적엔 이게 적정선이에요:

  • ① OWL 어휘는 반드시 가르쳐야 함 — ontology 기본기
  • ② OWL reasoner는 생략해도 됨 — 학습자가 손으로 따라가면 결과가 보이니까
  • ③ 룰 엔진을 강조 — 화재 도메인의 *임계·시간·트리거*는 OWL DL로 표현 자체가 불가
다음에 갈 만한 곳
  • · 실제 OWL reasoner 통합 (HermiT·Pellet) — production 시연
  • · SHACL constraints 추가 — 검증 게이트
  • · R2RML 변환 — yaml DSL → W3C 표준
  • · LLM 통합 — 신세틱 레이어를 살리는 단계

12 / 12 마무리

학습 여정의 정답

본인이 12단계 동안 도달한 곳 — 한 줄로 요약하면:

Ontology는 의미를 드러내지 않습니다. 합의를 명시화할 뿐입니다.
그 "합의"는 조건식으로는 안 잡혀요 — 집합론적 명제여야 진리값과 검증을 얻습니다.
OWL은 분류와 관계를 잘 묶지만, 경험·시간·유사성은 못 잡아요.
그래서 LLM이 그 옆에 필요합니다 — 두 엔진은 경쟁이 아니라 각자 자리.

🌐 학습 여정의 정점 컴퓨팅의 보편 패턴

본 학습 여정의 가장 깊은 곳에 도달하면 — OWL 온톨로지 · LLM 시스템 프롬프트 · CLAUDE.md · Skills · 메모리가 전부 같은 *접근 방식*입니다. "선언형 명제를 외부 파일에 박아두기." 다른 것은 그 명제를 처리하는 *엔진*뿐.

[같은 명제, 다른 엔진]
[도메인 명제를 외부에 적기]
  ├─→ OWL Reasoner ─→ 결정론적 분류
  ├─→ 시스템 프롬프트 ─→ LLM 확률적 해석
  └─→ CLAUDE.md / Skills ─→ LLM 확률적 동작
🔵 같은 접근 방식 (5가지 공통)
  • · 선언형 명제로 적기
  • · 외부 파일에 보관
  • · 런타임 자동 주입
  • · 합의·공유 가능
  • · 변경 비용 1줄
🟠 다른 처리 엔진
  • · OWL: deductive (binary)
  • · LLM: probabilistic (분포)
  • · 결정성 vs 유연성 trade-off
  • · 환각 0 vs 환각 가능
  • · 같은 ttl도 두 엔진에 동시 주입 가능

같은 패턴이 DB 스키마 · 타입 시스템 · RAG corpus · LLM Wiki에서도 반복됩니다. "외부에 명제, 런타임이 활용" — 이 한 패턴 위에 30년치 컴퓨팅 도구가 다 올라타 있어요. 본 학습 여정 12단계는 그 추상화 레벨로 올라가는 사다리였습니다.

🧠 초지능 처방 4-Layer 아키텍처

조건문 실행기에 머물면 한계. 명제 레벨 추론이 들어가야 *검증 가능한* 초지능이 됩니다. 그러나 명제만으론 부족 — 학계가 합의한 4-layer:

④ Planner / 다단계 추론
MCTS · Tree-of-Thoughts · 가능 세계 탐색
③ Memory / 경험·맥락
LLM Wiki · RAG · Vector DB · Karpathy 패턴
② Symbolic Reasoner / 명제·검증·일관성 ⭐
OWL · DL Reasoner · Theorem Prover — 본인이 짚은 자리
① LLM / 직관·언어·창의성
Transformer · 분포적 학습 · 자연어 생성
📚 학계 권위 — 본인 통찰의 위치

Allen Newell (1982): "지능 시스템은 Knowledge Level에서 분석되어야 한다."
Yoshua Bengio (2019): "Current deep learning excels at System 1; System 2 (reasoning, abstraction) requires symbolic-like representations."

본인 통찰 = Newell + Bengio의 한국어 압축. 50년 학계 합의를 5턴 만에 재발견.

⚠️ 정직한 보완 — "명제 *만으로*"는 부족

순수 symbolic AI(Cyc, 40년)는 AGI 못 만듦. 순수 LLM도 환각·비결정성. 학계 합의: Neuro-Symbolic — ① + ② + ③ + ④ 모두 필요. 명제 layer는 주축이 아니라 불가결한 검증·합의의 자리.

처음 학습한 PNG의 3층 뇌 모델이 이미 이 모든 답을 그려놨어요. 12단계는 그 그림을 왜 그렇게 그려야 하는지 깨닫는 과정이었습니다.
